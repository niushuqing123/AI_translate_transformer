# 项目设计说明

## 1. 为什么选择 Multi30k

这个项目面向的是“**课程作业 + 快速跑通 + 低显存**”场景，而不是追求竞赛级效果。Multi30k 有几个优势：

1. 数据体量小，训练集只有约 2.9 万句对；
2. 文本较短，平均句长低，显存压力更小；
3. 英德翻译任务经典、资料多，适合写实验报告；
4. 下载和处理都简单，适合本地闭环脚本化。

## 2. 为什么使用 Tiny Transformer

标准 Transformer-base 对课程作业来说往往偏重，显存和训练时间都不划算。本项目将模型缩小为：

- `d_model = 128`
- `nhead = 4`
- `encoder layers = 2`
- `decoder layers = 2`
- `dim_feedforward = 256`

这类配置虽然不追求最强效果，但更适合：

- 1GB 左右显存设备
- 需要完整跑通训练流程
- 需要做消融实验
- 需要生成可以写入实验报告的可复现实验结果

## 3. 为什么不依赖 BPE / SentencePiece

从工程角度看，BPE 能进一步改善翻译质量；但从课程作业角度看，它也会带来：

- 额外依赖
- 额外脚本复杂度
- 额外调试成本

因此本项目选择更轻量的方案：

- Unicode 友好的正则分词
- 训练集上构建词表
- 通过 `min_freq + vocab cap` 限制词表大小

这种方式牺牲了部分效果上限，但换来了：

- 代码更短、更清晰
- 更容易解释原理
- 更容易在答辩或课程检查时说明清楚

## 4. 低显存策略

本项目使用的低显存策略包括：

1. 小模型：2 层编码器 + 2 层解码器
2. 小批大小：`batch_size = 8`
3. 梯度累积：`grad_accum_steps = 4`
4. 最大长度裁剪：`max_len = 40`
5. CUDA 环境自动混合精度
6. 训练过程记录峰值显存

## 5. 对照实验设计逻辑

### 实验 1：Baseline

- 正弦位置编码
- 不使用标签平滑

作用：作为标准对照组。

### 实验 2：Learned Positional Encoding

- 将位置编码替换为可学习位置嵌入
- 其他超参数不变

作用：观察位置编码方式对小模型翻译效果的影响。

### 可选扩展：Label Smoothing

- 在基线配置上加入 `label_smoothing = 0.1`
- 其他参数保持一致

作用：如果你还有时间，再观察轻量训练技巧是否改善泛化。

## 6. 结果解释建议

建议在实验报告中关注以下维度：

- BLEU 是否提升
- chrF 是否提升
- 标签平滑是否牺牲训练集拟合、但提升测试集泛化
- 学习式位置编码是否比固定位置编码更适合该任务
- 翻译样例中常见错误类型：
  - 漏译
  - 词序错误
  - 词形变化错误
  - 未登录词替换为 `<unk>` 对应近似输出

## 7. 可扩展方向

若你后续还有时间，可继续扩展：

- 引入 beam search
- 增加 BPE 子词建模
- 对比不同层数/不同 `d_model`
- 更换为 IWSLT14 数据集
- 增加 attention 可视化
